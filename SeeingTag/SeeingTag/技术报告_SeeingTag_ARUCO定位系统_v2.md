# SeeingTag — 基于ARUCO标记的智能车实时定位系统技术报告

## 1. 项目概述

### 1.1 项目背景

本项目面向第21届全国智能汽车大赛人工智能模型组别，设计并实现了一套基于ARUCO视觉标记的智能车实时定位系统（SeeingTag）。系统通过单目摄像头采集赛道画面，识别布置在赛道固定位置的ARUCO标记码以及智能车上的标记码，利用Perspective-n-Point（PnP）算法解算摄像头和智能车的三维位姿，并通过UDP协议将位置数据发送至上位机（Unity），实现虚实融合的定位与AR显示。

### 1.2 系统目标

- 实时识别赛道上的ARUCO标记码（固定参考标记ID 1-4，车载标记ID 10）
- 通过多标记联合PnP位姿估计，计算摄像头在世界坐标系中的位姿
- 通过单标记PnP估算车载标签相对于摄像头的位姿，进而换算为世界坐标
- 通过UDP协议将智能车位置与偏航角数据发送至Unity AR环境
- 实现虚拟AR摄像头与真实智能车位置/姿态的同步

### 2.3 坐标系定义

```
世界坐标系 (右手系)：

    Z (赛道纵深方向，远离摄像头)
     ↑
     │
     │         X (赛道横向)
     │        ↗
     │       /
     │      /
     ──────●────────→ X (0~5m)
    /      │
   /       │
  Y (高度方向，向上为正)
```

赛道实际尺寸：**5m × 4m**（X方向0~5m，Z方向0~4m）

固定标签均布置于地面（Y=0），车载标签高度 Y=0.16m（固定值）。

## 3. 技术实现

### 3.1 ARUCO标记设计

采用 **DICT_4X4_50** 字典（4×4比特编码，50个标记），标记物理尺寸 **166mm × 166mm**。

**标记布局：**

| 标记ID | 世界坐标 (X, Y, Z) | 用途 |
|--------|-------------------|------|
| 1 | (0.0, 0.0, 0.0) | 固定参考点-左下角 |
| 2 | (5.0, 0.0, 0.0) | 固定参考点-右下角 |
| 3 | (5.0, 0.0, 4.0) | 固定参考点-右上角 |
| 4 | (0.0, 0.0, 4.0) | 固定参考点-左上角 |
| 10 | 车上 | 智能车位置标记 |

四个固定标签构成赛道矩形边界，标签间距在X方向为5m，Z方向为4m。

### 3.2 位姿估计算法：Perspective-n-Point (PnP)

#### 3.2.1 算法原理

PnP问题描述：已知一组**3D世界坐标点**及其对应的**2D图像投影点**，求解相机在世界坐标系中的位姿（旋转矩阵R和平移向量t）。

基本投影方程：

```
s · [u, v, 1]^T = K · [R | t] · [X, Y, Z, 1]^T
```

其中：
- (u, v)：图像像素坐标
- (X, Y, Z)：世界坐标
- K：相机内参矩阵
- R, t：相机外参（旋转矩阵和平移向量）
- s：尺度因子

#### 3.2.2 单标记PnP（用于车载标签）

对于单个ARUCO标记，已知其4个角点在**标签局部坐标系**中的坐标（以标签中心为原点）：

```
objPoints = [(-s2, -s2, 0), (s2, -s2, 0), (s2, s2, 0), (-s2, s2, 0)]
```

其中 `s2 = tag_size / 2 = 0.083m`。

在图像中检测到该标记的4个角点像素坐标后，调用 `cv2.solvePnP()` 求解标签相对于相机的位姿：

```python
ok, rvec, tvec = cv2.solvePnP(objPoints, imgPoints, K, None, flags=cv2.SOLVEPNP_ITERATIVE)
```

返回值 `rvec`（旋转向量）和 `tvec`（平移向量）描述了**标签坐标系 → 相机坐标系**的变换。

#### 3.2.3 多标记联合PnP（用于固定参考标签）

当画面中检测到2个或以上的固定标签时，将所有可见固定标签的4个角点**合并为一个3D-2D点集**，统一调用solvePnP求解相机位姿：

```python
for each fixed tag:
    wp = tag_world_position[tag_id]      # 标签中心世界坐标
    for each corner i (0~3):
        obj_pts.append(wp + local_corners[i])  # 角点世界坐标
        img_pts.append(image_corners[i])       # 角点像素坐标

ok, rvec, tvec = cv2.solvePnP(obj_pts, img_pts, K, None, flags=cv2.SOLVEPNP_ITERATIVE)
```

使用多个标签的角点增加了匹配点对数量，提高了PnP求解的**精度和鲁棒性**。

### 3.3 坐标转换与定位计算

#### 3.3.1 相机世界位姿

由多标记PnP直接获得相机在世界坐标系中的旋转向量 `rvec_cam` 和平移向量 `tvec_cam`：

```python
R_cam, _ = cv2.Rodrigues(rvec_cam)   # 旋转向量 → 旋转矩阵
```

#### 3.3.2 车载标签世界坐标

先由单标记PnP获得车载标签相对于相机的平移向量 `tvec_rel`，再转换到世界坐标系：

```python
car_pos_world = R_cam @ tvec_rel + tvec_cam
```

由于智能车的摄像头高度固定为0.16m，计算后强制将Y坐标设为0.16：

```python
car_pos = [car_pos_world[0], 0.16, car_pos_world[2]]
```

#### 3.3.3 偏航角计算

车载标签的朝向由相对旋转矩阵 `R_rel` 经世界旋转后得到：

```python
R_rel, _ = cv2.Rodrigues(rvec_rel)
R_car_world = R_cam @ R_rel
yaw = atan2(R_car_world[2, 0], R_car_world[0, 0])
```

偏航角为绕Y轴的旋转角，用于Unity中智能车的朝向显示。

### 3.4 低通滤波

为减少帧间抖动，对车载标签的平移向量 `tvec_rel` 进行一阶低通滤波（指数平滑）：

```python
state = alpha × current_value + (1 - alpha) × previous_state
```

滤波系数 `alpha = 0.3`，在**响应速度**和**平滑度**之间取得平衡。

### 3.5 相机内参

由于未进行正式的棋盘格标定，相机内参矩阵K通过视场角（FOV）估算：

```python
fx = width / (2 × tan(FOV_deg / 2))
fy = height / (2 × tan(FOV_deg / 2))
cx = width / 2
cy = height / 2
```

默认FOV = 100°，实际使用时需根据摄像头实际参数调整。

### 3.6 数据通信协议

采用JSON格式的UDP数据包发送至Unity端：

```json
{
  "type": "robot_position",
  "pos": [x, y, z],
  "euler": [0.0, 0.0, yaw],
  "seq": 序列号,
  "timestamp": 时间戳
}
```

**通信参数：**

| 参数 | 值 |
|------|-----|
| 目标IP | 10.61.148.125（可配置） |
| 端口 | 9005 |
| 协议 | UDP |
| 发送频率 | 与帧率一致（~15-30Hz） |
| 数据格式 | JSON字符串 |

### 4.2 核心类说明

#### TagTracker类

系统核心类，封装了标记检测、位姿估计、坐标转换、滤波、UDP发送、HUD绘制等功能。

**主要方法：**

| 方法 | 功能 |
|------|------|
| `detect(gray)` | 检测ARUCO标记，返回ID、角点、像素宽度 |
| `pnp_single(corners)` | 单标记PnP位姿估计 |
| `pnp_multi_fixed(detections)` | 多标记联合PnP位姿估计 |
| `locate(detections)` | 综合定位：计算相机位姿→车载标签世界坐标→偏航角 |
| `send_position(pos, yaw)` | 通过UDP发送位置数据 |
| `draw_hud(frame, ...)` | 在画面上绘制标签框、坐标轴、HUD信息 |
| `set_intrinsics(fov, w, h)` | 根据FOV和分辨率设置相机内参 |

#### UdpSender类

封装UDP通信，每次发送自增序列号和时间戳，方便Unity端校验数据连续性。

#### LowPassFilter类

实现基于字典的键值对低通滤波，每个标签ID独立维护滤波状态。

## 5. 实验结果与问题分析

### 5.1 定位功能验证

- ✅ **ARUCO标记检测**：成功检测ID 1-4固定标签和ID 10车载标签
- ✅ **多标记PnP**：使用2个以上固定标签可稳定计算相机位姿
- ✅ **坐标转换**：成功将车载标签相对位姿转换为世界坐标
- ✅ **UDP通信**：位置数据成功发送至目标IP

### 5.2 存在的问题

#### 5.2.1 摄像头延迟问题

使用手机IP摄像头App通过RTSP协议传输时，实测延迟高达 **5~30秒**，主要原因：

| 原因 | 说明 |
|------|------|
| WiFi网络传输延迟 | RTSP流经WiFi传输，编码/解码/网络缓冲引入延迟 |
| 编码缓冲 | 手机App的视频编码器内部缓冲区较大 |
| OpenCV读取缓冲 | `cv2.VideoCapture` 内部维护帧队列，累积延迟 |

**解决方案：**
- 更换DroidCam等低延迟App（WiFi延迟约100-200ms）
- 使用USB连接手机（延迟<50ms）
- 更换USB广角摄像头直连电脑（推荐，延迟<30ms）

#### 5.2.2 画面畸变问题

手机广角镜头（尤其是超广角模式）存在明显**桶形畸变**，导致：
- 画面边缘标签角点位置偏移，PnP精度下降
- 大视角下四个标签难以同时清晰识别

**解决方案：**
- 使用畸变较小的镜头（如尼康Z5 + 16mm镜头 + HDMI采集卡）
- 进行相机标定，获取畸变系数后用 `cv2.undistort()` 校正
- 避免使用超广角模式

#### 5.2.3 标签识别覆盖问题

| 摄像头类型 | 分辨率 | 视野覆盖 | 标签识别 |
|-----------|--------|---------|---------|
| 普通USB摄像头 | 640×480 | 窄，只能看到1-2个标签 | 识别不全 |
| 手机广角 | 1280×720 | 能覆盖4个标签，但畸变大 | 边缘标签识别不稳定 |
| 尼康Z5+16mm+采集卡 | 1920×1080 | 覆盖4个标签，畸变小 | **最佳方案** |

### 5.3 定位精度分析

定位精度受以下因素影响：

| 因素 | 影响 |
|------|------|
| 摄像头分辨率 | 分辨率越高，标签角点检测越精确 |
| 标签物理尺寸 | 166mm标签在画面中占比越大，PnP精度越高 |
| 标签数量 | 使用更多固定标签可提高相机位姿估计精度 |
| 相机内参准确性 | 估算的FOV与实际值偏差会影响绝对定位精度 |
| 画面畸变 | 未校正的畸变会导致PnP解算偏差 |

## 6. 后续优化方向

### 6.1 相机标定

正式标定相机内参（焦距、光心、畸变系数），替代当前基于FOV的估算方法：

```python
# 使用棋盘格标定
ret, K, dist_coeffs, rvecs, tvecs = cv2.calibrateCamera(objpoints, imgpoints, gray.shape[::-1], None, None)

# 畸变校正
frame_undistorted = cv2.undistort(frame, K, dist_coeffs)
```

### 6.2 硬件升级

- **USB广角摄像头（1920×1080以上）**：替代手机RTSP方案，解决延迟和畸变问题
- **尼康Z5 + HDMI采集卡**：利用高质量镜头获得低畸变、高分辨率画面

### 6.3 算法优化

- **自适应滤波**：根据检测置信度动态调整滤波系数
- **异常值剔除**：对PnP结果进行合理性校验（如位置是否在赛道范围内）
- **多帧融合**：利用历史帧信息平滑定位结果

### 6.4 实时性优化

- 降低检测分辨率（如先缩小图像检测标签，再在原图上画框）
- 使用GPU加速OpenCV操作
- 优化PnP求解策略（IPPE_SQUARE比ITERATIVE更快）
